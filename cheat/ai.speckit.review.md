
- **research.md** — Start here. This is the Phase 0 output where the agent explored libraries, patterns, and trade-offs. If it picked the wrong Hex packages or made bad assumptions about your stack (e.g., suggesting Absinthe when you want REST, or recommending a Node library instead of an OTP solution), everything downstream is built on a shaky foundation. Catch it early.
- **data-model.md** — Your Ecto schemas and relationships are the backbone of any Phoenix app. Verify the associations, field types, and migration ordering make sense. Check that it's using proper Ecto conventions (embedded schemas vs separate tables, timestamps, etc.). A bad data model is expensive to fix after implementation starts.
- **contracts/** — Review the API contracts next, since they sit on top of the data model. Confirm the endpoints, request/response shapes, and status codes align with how you'd structure your Phoenix router and controllers (or your LiveView surface if it's not a REST API).
- **plan.md** — Read this last as the overarching document. By now you've already validated the building blocks, so you can focus on the higher-level architecture decisions: context boundaries, supervision trees, deployment concerns, and the overall flow. You'll spot inconsistencies faster because you already know what the individual pieces look like.
- **quickstart.md** — This is more of a sanity check. Skim it to make sure the setup instructions actually match the plan (correct Mix tasks, dependencies in `mix.exs`, config files, etc.).
